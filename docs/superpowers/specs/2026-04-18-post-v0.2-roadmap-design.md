## 0. 背景与动机

v0.2.0 已落地的基线功能见 `CHANGELOG.md` 与 `docs/ARCHITECTURE.md`。审计后发现三类未完成项:

1. **显式 deferred**(CHANGELOG 已登记):AI Diff 卡、BGE 神经嵌入、Cache Components、React Compiler
2. **类型已声明但未实现**:`AnalysisArtifact.kind: 'arc-map'`、`AIUsageEvent` 数据落地但无 UI
3. **隐性技术债**(本次审计发现):`use-chapter-generation.ts` 直接用 `fetch` 调 OpenAI 协议,**完全绕过** v0.2 统一的 `streamChat()` AI 层 —— Anthropic provider 下不可用,无 caching、无 tool use、无 hybrid RAG

本路线图聚焦**收尾 v0.2 遗留**,不引入新方向。AI 用量面板、桌面端封装、新 AI 工具(改写/续写/对话扮演)明确推迟到 v0.5+。

## 2. v0.3.0 详细设计

### F1 · AI Diff 卡

#### 2.1.1 问题

两条 AI 产出路径目前均为「一键塞进编辑器」:
- `use-ai-chat` 的草稿 → `editorRef.current?.insertText(content)`(见 `src/app/projects/[id]/page.tsx:174`)
- `use-chapter-generation.acceptContent()` → 直接 `updateChapterContent`,多段落被压成单个 paragraph(见 `src/lib/hooks/use-chapter-generation.ts:243-252`)

无法审阅、无法局部接受、破坏段落结构。

#### 2.1.2 交互模型

统一三按钮:**接受** · **拒绝** · **重新生成**。局部 hunk 级接受推迟到 v0.5+。

- **行内 diff 卡**:替换聊天气泡里的「采纳到编辑器」按钮
- **全屏 diff 视图**:替换生成面板的 CardFooter

#### 2.1.3 Diff 计算

新增 `src/lib/diff/text-diff.ts`。依赖 npm [`diff`](https://www.npmjs.com/package/diff) 包的 `diffChars`(字符级 diff;中文无词边界,`diffWords*` 会降级为行级,不可用)。不选 `diff-match-patch`:多 ~40KB 包体积无正当理由。

输出契约:

```ts
type DiffSegment = {
  type: 'equal' | 'insert' | 'delete'
  text: string
}
function computeDiff(before: string, after: string): DiffSegment[]
```

#### 2.1.4 组件

- `src/components/ai/diff-card.tsx` — 紧凑行内版,红/绿高亮 + 三按钮
- `src/components/ai/diff-view.tsx` — 全屏版,支持左右并排 / 上下对齐切换

#### 2.1.5 接受逻辑

统一函数 `applyAIDraft(editorHandle, diff, { createRevision: true })`:
1. 计算合并后 Tiptap 文档 → `editorHandle.setContent(newDoc)`
2. 创建 `source: 'ai-draft'` 版本快照(revisions 表已支持)
3. 持久化到当前章节

#### 2.1.6 hooks 改动

- `use-ai-chat.ts` — 新状态 `pendingDrafts: PendingDraft[]`,替换现有 `hasDraft` 字段的直接消费路径
- `use-chapter-generation.ts` — 移除 `acceptContent`,改为 `getPendingDiff()` 返回给 UI

#### 2.1.7 顺手修复

章节生成的多段落丢失:AI 流返回文本按 `\n\n` 分段为多个 `paragraph` 节点,而非单一 paragraph。

#### 2.1.8 测试

- `text-diff.test.ts` — 中文字符 diff 正确性
- `diff-card.test.tsx` — 按钮行为 + revision 创建
- Playwright e2e:聊天草稿 → diff 卡 → 接受,验证编辑器内容与 revision 同步更新

#### 2.1.9 Scope 外

- 局部 diff hunk 部分接受
- 三方合并(编辑器期间的本地变更 vs AI 草稿)

### F2 · arc-map 剧情弧线分析

#### 2.2.1 数据模型

复用 `AnalysisArtifact` 表,`kind: 'arc-map'`。`invalidationKey` 计算:按 `order` 排序的活跃章节 → 每章用 `extractPlainText(chapter.content)` 取纯文本 → 作为数组传入 `contentHash(texts)` 得到单一哈希字符串(二者均已由 `src/lib/analysis/content-hash.ts` 导出)。`result` 字段 schema:

```ts
type ArcMapResult = {
  chapters: {
    chapterId: string
    beat: 'setup' | 'rising' | 'climax' | 'falling' | 'resolution'
    tension: number       // 0..10
    emotion: number       // -1..1
    keyCharacters: string[]  // WorldEntry id
    keyEvents: string[]      // 短描述
    summary: string          // ≤30 字
  }[]
  overallArc: string         // ≤120 字
}
```

#### 2.2.2 生成方式

一次 Claude 调用 + **tool use** 强制结构化。新增 `emit_arc_map` tool(`src/lib/ai/tools/schemas.ts`),schema 对应上述类型。系统提示词中注入全部章节摘要 + RAG 命中的世界观条目。

长项目(~50k+ token)用批处理:每批 10 章,最后合成 `overallArc`。

#### 2.2.3 新模块

- `src/lib/analysis/arc-map.ts` — `generateArcMap(db, projectId, { config }): Promise<ArcMapResult>`,含缓存命中判断
- `src/components/analysis/arc-map-view.tsx` — 网格可视化:每章一行,标题 + 张力条 + 情绪色点;悬停显示 summary;点击跳章节;顶部横幅显示 `overallArc`

**故意不做**:折线图、SVG 动画、复杂交互 —— 这是分析卡不是游戏化页面。

#### 2.2.4 集成点

`/projects/[id]/analysis/page.tsx` 新增 tab「剧情弧线」,与现有「关系图」「时间线」「文风」并列。

#### 2.2.5 Provider 降级

OpenAI 兼容 provider 不保证 tool use。检测到 `provider === 'openai-compatible'` 时显示占位卡「剧情弧线分析需要 Anthropic provider」,不暴露按钮。理由:无 schema 保底的剧情分析完全不可靠,不如不给。

#### 2.2.6 测试

- `arc-map.test.ts` — mock AI 调用,覆盖缓存命中/失效、短/长项目两条路径
- `arc-map-view.test.tsx` — 渲染 + 空数据态

#### 2.2.7 Scope 外

- 按角色视角的人物弧线 —— v0.5+
- 自动建议「张力过低」型提示 —— 侵入性强,按用户反馈决定

### F3 · 生成面板重写

#### 2.3.1 当前问题(code evidence)

- `use-chapter-generation.ts:123`:直接 `fetch('${config.baseUrl}/chat/completions')`,强绑定 OpenAI 协议 —— **Anthropic provider 下不可用**
- 无 prompt caching
- 无 tool use
- 用老关键词 `extractKeywords` 而非 hybrid RAG,中文指代性查询召回差
- `acceptContent:243-252` 将多段文本压成单 paragraph,丢失所有换行

#### 2.3.2 重写方案

`use-chapter-generation.ts` 彻底重写为 `streamChat()` 调用者:

```ts
function useChapterGeneration(projectId, chapterId) {
  const config = useAIConfig(projectId)
  const { entries } = useWorldEntries(projectId)

  async function start() {
    const relevantEntries = await searchRelevantEntries({
      db, projectId, embedder, query: outlineSummary, entries, topK: 20
    })
    const segmentedSystem = buildGenerationSystemPrompt({
      chapterTitle, outlineSummary, targetWordCount,
      worldBibleContext: relevantEntries
    })
    const events = streamChat(config, { segmentedSystem, messages: [...], signal })
    for await (const event of events) {
      if (event.type === 'text_delta') setStreamingContent(s => s + event.delta)
      else if (event.type === 'usage') recordUsage(event)
      else if (event.type === 'error') setError(event.message)
    }
  }
}
```

#### 2.3.3 prompt 分段(cache 断点)

- **BASE**:生成器通用指令(网文风格、段落结构、悬念)—— 缓存
- **worldBibleContext**:hybrid RAG 命中的条目 —— 缓存
- **runtimeContext**:章节标题 / 大纲摘要 / 目标字数 —— 不缓存

#### 2.3.4 多段落保留

`splitIntoParagraphs(text)` 按 `\n\n` 分段。每段一个 Tiptap `paragraph` 节点。空行处理规范化;Markdown 列表降级为纯文本(不向编辑器塞非预期节点)。

#### 2.3.5 与 F1 的衔接

移除 `acceptContent`。产出走 F1 的 `diff-view`(全屏模式)。生成面板的 CardFooter 三按钮替换现有「采纳到编辑器」。

#### 2.3.6 Provider 行为

- Anthropic:完整链路,预期 cache 命中率 >90%
- OpenAI 兼容:同样走 `streamChat()`(v0.2 已统一 provider 抽象),无 tool use / caching,但不再是直接 fetch

#### 2.3.7 AI 用量记录(数据先行)

本轮不做用量 UI。但 `streamChat` 的 `type: 'usage'` 事件必须**写入 `aiUsage` 表**。v0.5 做面板时即有历史数据。

#### 2.3.8 测试

- `use-chapter-generation.test.ts` — mock `streamChat`,覆盖两条 provider 路径、取消、错误、多段落解析
- Playwright e2e:配大纲 → 生成 → 看到 diff-view → 接受

#### 2.3.9 Scope 外

- 章节续写(从已有末尾延展)—— v0.5+
- 批量生成多章 —— v0.5+

## 3. v0.4.0 详细设计

### F4 · BGE 神经嵌入

#### 3.1.1 目标

用 `BGEEmbedder`(`bge-small-zh-v1.5`,512 维)替换 `HashGramEmbedder`。`Embedder` 接口在 v0.2 已稳定。保留 hash-gram 作为永远可用的回退。

#### 3.1.2 新模块

`src/lib/rag/bge-embedder.ts`:

```ts
class BGEEmbedder implements Embedder {
  readonly dim = 512
  readonly id = 'bge-small-zh-v1.5'
  private pipeline: FeatureExtractionPipeline | null = null

  async embed(texts: string[]): Promise<Float32Array[]> {
    if (!this.pipeline) {
      this.pipeline = await pipeline(
        'feature-extraction',
        'Xenova/bge-small-zh-v1.5'
      )
    }
    // batch + L2 normalize
  }
}
```

依赖:`@xenova/transformers`(~3MB npm)+ 模型文件(~100MB,HF CDN + browser Cache API,transformers.js 内置 IndexedDB 持久化)。

#### 3.1.3 首次加载 UX(关键)

新组件 `EmbedderSetupCard`(仅在 AI 配置对话框内):
- **未下载**:按钮「启用高精度中文检索(约 100MB 一次性下载)」
- **下载中**:进度条 + 已下载 MB + 取消按钮
- **已下载**:✅ 已启用,可切回 hash-gram
- **失败**:降级提示,自动回退 hash-gram

**默认不自动下载**,用户主动点才下。App 开箱即用仍是 hash-gram。

#### 3.1.4 偏好持久化

`localStorage.setItem('inkforge-embedder', 'bge' | 'hashgram')`。`default-embedder.ts`:

```ts
async function getDefaultEmbedder(): Promise<Embedder> {
  const pref = localStorage.getItem('inkforge-embedder')
  if (pref === 'bge' && await isBGEAvailable()) return new BGEEmbedder()
  return new HashGramEmbedder()
}
```

#### 3.1.5 embedderId 切换

`indexer.ts` 已处理 `embedderId` 变更触发全量 reindex,只需补测试验证。

#### 3.1.6 性能基准(决策证据)

新脚本 `scripts/benchmark-rag.ts`:在 500 条目样本项目上跑 20 条中文指代性查询(「她的师父」「那把剑」),对比 hash-gram vs BGE 的 Top-5 命中率。结果写 `docs/RAG-BENCHMARK.md`,作为是否鼓励用户启用的决策依据。

#### 3.1.7 包体积控制

- `@xenova/transformers` 只通过**动态 import** 加载,首屏 bundle 零增加
- 未启用 BGE 的用户永远不下载模型文件

#### 3.1.8 测试

- `bge-embedder.test.ts` — mock transformers.js,测接口契约 + L2 归一化
- `default-embedder.test.ts` — 三条路径(无偏好 / 偏好 BGE 可用 / 偏好 BGE 不可用)
- 首次下载的 smoke 走 Playwright 手动验证(jsdom 跑不起 transformers.js 的 WASM)

#### 3.1.9 Scope 外

- 句子级嵌入(更细粒度 chunking)—— v0.5+
- multilingual-e5 或其他模型 —— 按用户反馈

### F5 · Cache Components + F6 · React Compiler

#### 3.2.1 前置依赖

Next.js 16.2.3 的 `NextConfig` 类型不识别 `experimental.cacheComponents` 和 `experimental.reactCompiler`(CHANGELOG 已登记)。

- **路径 A(默认)**:等 Next.js 16.3 stable。当前 canary 已就绪,按官方节奏 ~4-6 周后 stable
- **路径 B**:上 canary + `@ts-expect-error` 压类型报错。**不推荐** —— canary breaking change 风险,零用户可见价值不值得

若 16.3 stable 在 v0.3 发布后仍未就绪,v0.4 延期,期间做 v0.3 的用户反馈迭代。

#### 3.2.2 Cache Components 启用范围

InkForge 是 IndexedDB 驱动的客户端应用,大部分组件必须 `'use client'`,CC 收益集中在 server 层外壳:

- `src/app/page.tsx`(Dashboard):数据来自客户端 IndexedDB,**用不上**
- `src/app/(authenticated)/layout.tsx`:Supabase 会话校验,可 `"use cache"`
- `src/app/projects/[id]/layout.tsx`:若有 server-fetch 的项目元数据镜像,可 `"use cache"`
- `src/proxy.ts`:已是 server 层,无需改

**审视后若 cacheable 路径 <3 处,坦诚写下「Cache Components 对当前架构价值低,本次不启用」**,不为了用而用。

#### 3.2.3 React Compiler 启用

- 全局 `experimental.reactCompiler: true`
- ESLint 启用 `eslint-plugin-react-compiler`(强制,编译器对违规组件 bail out)
- **不主动删除**现有 `useMemo` / `useCallback`。编译器接管后它们变冗余但无害;下一轮再按模块清理,避免本轮 diff 过大
- CI 监控:每次 build 日志里抓 "bailed out" 数量,写入 artifact

#### 3.2.4 性能基准

Playwright 测:首屏 LCP、100 章节工作区打开时间、打字输入 INP。启用前后对比,写 `docs/PERF-BASELINE.md`。

#### 3.2.5 失败回退

两 flag 均在 `experimental` 下。Config 里 `false` 即关。v0.4 发布后可热回滚。

#### 3.2.6 测试

- Playwright 回归套件全量通过(核心验证是「没 break」)
- `docs/PERF-BASELINE.md` 作为性能验证证据

#### 3.2.7 Scope 外(写入 v0.4 CHANGELOG deferred)

- `useMemo` / `useCallback` 批量清理 —— 下一轮
- Server Actions 重写 —— 下一轮
- Turbopack build —— 已在(默认)

## 4. 跨版本事项

### 4.1 文档

每个版本发布时:
- 更新 `CHANGELOG.md`(v0.3.0、v0.4.0 各一段)
- `README.md` 的亮点列表按需补充
- v0.3 触及 diff / 生成 → 更新 `docs/AI-LAYER.md`
- v0.4 触及 BGE → 更新 `docs/RAG.md` 并新增 `docs/RAG-BENCHMARK.md`
- v0.4 触及 framework → 新增 `docs/PERF-BASELINE.md`

### 4.2 测试门槛

- 每个特性必须有单测 + 至少一个 Playwright e2e
- v0.4 发布前现有 101 单测 + e2e 全部通过
- 无新增 ESLint 警告(`react-compiler` 规则启用后尤其严格)

### 4.3 明确不做(防 scope creep)

本轮路线图**不包括**以下,它们有价值但应留到 v0.5+:
- AI 用量面板 UI
- 桌面端封装(Tauri / Electron)
- 新 AI 工具(改写 / 续写 / 对话扮演 / 命名生成器)
- 局部 diff hunk 接受
- 人物弧线分析
- 句子级嵌入
- 社区模板 / 分享
- 付费/订阅模型

## 5. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| Next.js 16.3 stable 延期 | v0.4 延期 | v0.3 打磨期吸收;最坏情况下 F5/F6 推迟到 v0.5 |
| BGE 下载对部分地区用户不稳定 | 用户启用失败 | 保留 hash-gram fallback 永远可用;失败自动降级 |
| `diff` npm 包对中文字符边界处理不理想 | diff 卡显示错位 | `text-diff.test.ts` 建立中文 fixture 回归;若不足切 `diff-match-patch`(代价 ~40KB) |
| arc-map Claude 调用超 token 预算 | 长项目失败 | 实现即带 batch 模式(每批 10 章),非可选 |
| 生成面板重写引入回归 | 用户正在用的流程坏掉 | Playwright e2e 覆盖完整生成链路;保留旧 hook 一个 commit 周期以便快速回滚 |
