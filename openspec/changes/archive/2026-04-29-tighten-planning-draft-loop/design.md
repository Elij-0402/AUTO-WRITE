## Context

The repository now has the main planning layers in place: ideas, arcs, chapter plans, and scene cards. Scene cards can already be authored manually, generated by AI, stored in IndexedDB, and consumed by the chapter draft outline builder.

The remaining gap is not raw capability, but handoff fidelity. `ChapterDraftPanel` currently pre-fills from the linked chapter plan once and keys refresh mostly on chapter-plan timestamps plus scene-card count. That means scene-card content edits or reordering can leave the draft outline stale. At the same time, the current browser test only validates chapter-plan summary prefill; it does not prove that scene cards survive the full planning-to-draft path.

## Goals / Non-Goals

**Goals:**
- Make chapter-draft prefill reflect the latest linked planning state, including scene-card content and order.
- Prevent silent overwrite of a draft outline once the user has manually edited it.
- Make newly applied AI scene cards obvious in the planning workbench so users land on the new material.
- Add workflow-level regression coverage for scene-card-aware draft prefill.

**Non-Goals:**
- Redesign the drafting panel or planning workbench layout.
- Change the scene-card schema or add world-entry picking in this pass.
- Replace the current draft generation pipeline or prompt architecture.

## Decisions

### 1. Track planning freshness with a stable scene-aware fingerprint

The draft panel should not decide freshness from scene count alone. It should derive a fingerprint from the linked chapter plan plus the linked scene-card set, including IDs, order, and `updatedAt`, so content edits and reorders are visible to the handoff layer.

Alternative considered: rely only on `chapterPlan.updatedAt`. Rejected because scene-card mutations do not update the parent chapter plan and would still leave the outline stale.

### 2. Separate "prefill source changed" from "user edited outline"

The draft panel should keep two pieces of state: the latest planning fingerprint it has applied and whether the outline has been manually edited since that apply. If planning changes arrive before the user edits, the panel can refresh automatically. If the user has already edited the outline, the panel should show a lightweight refresh affordance instead of replacing content immediately.

Alternative considered: always overwrite the outline on planning changes. Rejected because it would destroy in-progress manual draft shaping.

### 3. Surface the first newly created scene after AI apply

When AI-generated scene cards are accepted, the workbench should expand or focus the first newly created scene card. This keeps the user in the generated result instead of leaving them on an older expanded scene.

Alternative considered: keep the current chapter-level reselection only. Rejected because it technically refreshes the list but does not reliably reveal the new beats when older scene cards already exist.

### 4. Add browser coverage at the real handoff boundary

The end-to-end regression should prove that linked chapter plans with scene cards prefill the draft panel in scene order and that refreshed planning data reaches drafting. This belongs in Playwright because the failure mode is cross-surface state handoff, not a pure helper bug.

Alternative considered: rely only on component tests around `buildDraftOutlineFromPlanning`. Rejected because those tests do not exercise the actual workspace mode switch and panel state behavior.

## Risks / Trade-offs

- [Risk] Auto-refresh rules may feel inconsistent if they are too implicit. -> Mitigation: only auto-refresh before local edits, then switch to an explicit refresh affordance.
- [Risk] A scene-aware fingerprint may cause extra draft-panel rerenders. -> Mitigation: keep the fingerprint small and derived from already-loaded snapshot data.
- [Risk] E2E coverage can become brittle if it depends on AI output. -> Mitigation: seed planning data directly and keep the assertions focused on visible handoff state.

## Migration Plan

No data migration is required.

1. Update draft-panel freshness logic and refresh controls.
2. Update planning AI apply behavior to reveal newly created scenes.
3. Extend unit/component coverage around refresh and focus behavior.
4. Add Playwright coverage for scene-card-aware draft prefill.

Rollback is low risk because the change is confined to UI state logic and tests.

## Open Questions

- Should the refresh affordance replace only the outline text, or also resync draft title and target word-count preset when linked planning changes?
- If AI-generated scene cards are applied to a chapter that already has manual scene cards, should the first newly created scene become expanded unconditionally, or only when no manual scene is currently being edited?
