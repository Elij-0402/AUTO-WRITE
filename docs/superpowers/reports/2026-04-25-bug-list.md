## Bug 清单

### BUG-001: 离线优先承诺被破坏 — 未登录用户被强制重定向到 /auth

- **模块**: 全局 / 认证
- **严重**: P0
- **状态**: open
- **来源**: e2e（次生：所有 E2E 失败）

**复现**：
1. 启动 `pnpm dev`
2. 用未登录浏览器访问 `http://localhost:3000/`
3. 期望：渲染 ProjectDashboard / EmptyDashboard
4. 实际：被 `src/proxy.ts` 重定向到 `/auth`，看到登录表单

**根因**：`src/proxy.ts:14-18`（D-49 设计）拦截所有未授权请求重定向到 /auth，但 `README.md` 与 `CLAUDE.md` 都明确产品定位是 **"离线优先、BYOK、数据本地、Supabase 可选"**。`(authenticated)` route group 实际只调 `createClient()` 不检查 user，`/projects/[id]` 也不在 (authenticated) 下，业务代码（`use-projects.ts`）已经做好了 user 为 null 的处理。proxy.ts 的拦截是孤立的、违反产品承诺的设计。

**期望**：未登录用户可以正常使用所有本地功能；只有用户主动启用云同步（如点击"登录同步"）才进入 /auth。

**实际**：所有路径（除 /auth、/api）被强制要求登录。

**截图**：见 `test-results/project-crud-create-projec-7163b-board-and-land-on-workspace-chromium/error-context.md` 中的 page snapshot。

**影响**：
- 阻塞所有 E2E 测试（8 个 failed）
- 阻塞匿名 / 未配置 Supabase 用户使用 InkForge
- 与产品定位严重不符

**产品决策**：按 README 路线修复（README 是用户承诺，优先级高于内部 spec D-49）。

### BUG-005: TypeScript 类型错误阻塞生产构建

- **模块**: 构建 / AI
- **严重**: P0
- **状态**: fixed
- **来源**: build
- **修复 commit**: `fix(ai): correct SegmentedSystemPrompt shape in relation-recommendation.ts`

**复现**：
1. 运行 `pnpm build`
2. `pnpm test` 通过（263 passed）
3. 构建在 TypeScript 类型检查阶段失败

**根因**：`src/lib/ai/relation-recommendation.ts:63-67` 使用了旧的 `segments` 字段构造 `SegmentedSystemPrompt`，但当前 `prompts.ts` 定义的 `SegmentedSystemPrompt` 接口只有 `baseInstruction | worldBibleContext | runtimeContext | chapterDraftContext` 四个字段，无 `segments` 字段。同样的问题可能存在于 `summarize.ts` 的 `SUMMARY_SYSTEM`。

`prompts.ts` 近期重构后接口已变更，但调用方未同步更新。

**影响**：
- `pnpm build` 在 TypeScript 阶段失败
- 生产构建无法完成
- 潜在类型问题影响 AI 功能正常工作

**修复方向**：
1. 修改 `relation-recommendation.ts` 使用正确的 `SegmentedSystemPrompt` 结构
2. 检查 `summarize.ts` 是否有同类问题
3. 同步更新所有调用方

### BUG-004: 项目工作区页面返回 500（Jest worker exhaustion）

- **模块**: 全局 / 开发服务器
- **严重**: P0
- **状态**: open
- **来源**: browser-qa

**复现**：
1. 启动 `pnpm dev`
2. 点击"开始第一个故事" → 填写标题 → 创建项目
3. 成功跳转 `/projects/[id]`，但页面返回 500 错误
4. 所有后续访问 `/projects/[id]` 均返回 500

**根因**：`next dev` 内部使用 Jest worker 来渲染 RSC Payload。项目创建后 RSC 请求 `?_rsc=lquux` 触发 Jest worker 异常：`"Jest worker encountered 2 child process exceptions, exceeding retry limit"`。Worker 池被耗尽后无法恢复，导致该 project 路由下所有请求返回 500，直到重启 dev server。

**影响**：
- 所有已创建的项目（ID `iNtxgms0f3yzeFI54m7Mf`）的工作区页面无法访问
- 新建项目后立即无法进入工作区
- E2E 测试中 project workspace 路由也会受影响

**截图**：`/.tmp-qa/02-project-workspace.png`

**修复方向**：
1. 在 Next.js 16 dev 中禁用 RSC Payload 的 Jest worker 渲染，或配置 `experimental.workerThreads: false`
2. 或者在 `next.config.ts` 中调整 jest worker 配置
3. 或者在 `next dev` 前设置 `NODE_OPTIONS='--max-old-space-size=...'` 避免 worker 溢出
4. 根本解决：分离 SSR 和测试 worker 池，或升级 Next.js patch

### BUG-002: E2E `project-workflow.spec.ts` createProject helper 用了错误按钮文案

- **模块**: 测试
- **严重**: P1
- **状态**: open
- **来源**: e2e

**复现**：
1. BUG-001 已修
2. 跑 `pnpm test:e2e --project=chromium`
3. 实际：`tests/e2e/project-workflow.spec.ts` 5 个测试全 timeout，等 `新建项目` 按钮
4. 期望：与其它 spec 一致使用 `开始第一个故事`（EmptyDashboard 实际文案）

**根因**：`tests/e2e/project-workflow.spec.ts:9` 的 `createProject` helper 用 `getByRole('button', { name: '新建项目' })`，但首次访问 `/` 时 IndexedDB 为空，渲染的是 `EmptyDashboard`（按钮文案 `开始第一个故事`，见 `src/components/project/empty-dashboard.tsx:34`）。其它三个 spec（`project-crud.spec.ts`、`chapter-write.spec.ts`、`world-bible.spec.ts`）用的是正确的 `开始第一个故事` 已通过。

`新建项目` 按钮只在已有项目时的 `ProjectDashboard` 中出现（`project-dashboard.tsx:151`）。

**修复方向**：让 `tests/e2e/project-workflow.spec.ts` 的 `createProject` 与其它 spec 一致 — 用 `开始第一个故事` 触发 modal。表单填写步骤也要同步对齐（其它 spec 用 `getByLabel(/标题/)`，这个 spec 用 `getByPlaceholder('输入书名')`，需检查 modal 实际控件）。

**进展**：createProject helper 已修（commit 待入），但同 spec 中 5 个 test 后续步骤同样过时 — 找不到 `新建章节`、`AI 输入框`、`新建角色` 等。整个 spec 与当前 UI 不同步，归并到 BUG-003。
