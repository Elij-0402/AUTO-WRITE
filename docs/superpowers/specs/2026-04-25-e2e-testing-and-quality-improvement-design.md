# InkForge 端到端测试与代码质量提升 - 设计规格书

Generated by brainstorming on 2026-04-25

## 1. 目标

对 InkForge 当前主分支进行一次全面的端到端测试与代码质量提升，覆盖：

- **B 阶段**：B3 穷举式 QA — 用真实浏览器跑通核心流程、边界场景、次要功能
- **C 阶段**：四项代码质量优化 — 性能、代码质量、可访问性、测试覆盖

最终交付一份带证据（截图、日志、before/after 数据）的总报告，以及一组有序的 commit。

## 2. 范围与不做的事

### In Scope

- 现有自动化测试（vitest 单元 + Playwright E2E）的运行与失败修复
- 用浏览器实测整套 InkForge 流程（项目 CRUD、章节编写、Tiptap 编辑器、AI 对话与建议、世界观条目与关系、分析页关系图、设置、Supabase 同步入口、导出、快捷键、移动端响应式）
- 性能 baseline 与优化（首屏、Tiptap 大文档、关系图渲染、bundle 体积）
- 代码质量（lint warnings、any 类型、重复代码、过长文件拆分）
- 可访问性 + DESIGN.md 合规（键盘导航、ARIA、对比度、字体/圆角/阴影/动画）
- 关键模块单元测试补全

### Out of Scope

- 新功能开发（不在 bug 范畴的 enhancement 仅记录到报告，不实施）
- 后端 / Supabase 服务端逻辑（项目是 BYOK + 本地优先，无自有后端）
- 大规模架构重构（拆分仅限于明显过长且影响阅读 / 修改可靠性的文件）
- DeepSeek 之外的 AI provider 实测（接口契约靠现有单测保证）

## 3. 执行模式

**模式 1 轻量版 + 检查点**：

1. 阶段 A 自动化测试 → 失败清单（不打断用户）
2. 阶段 B QA 实测 → bug 清单（不打断用户）
3. **检查点 1**：bug 清单出来后，简短同步给用户
4. 阶段 B 修复（自主推进）
5. 阶段 C 四个子项依次完成（自主推进）
6. **检查点 2**：阶段 C 开始前简短同步
7. 最终报告（一次性交付）

中途仅在以上两个检查点同步进度，不再频繁打断。

## 4. Commit 策略

- **同模块多 bug** → 合并为一个 commit（例：`fix(analysis): 修复关系图三个交互问题`）
- **跨模块 bug** → 独立 commit
- **C 阶段每个子项** → 独立 commit（perf / quality / a11y / test）
- 全程遵循项目现有 Conventional Commits 规范
- 不强行拆分 commit 让 git log 碎片化，也不堆成一个大 commit

## 5. 产品决策处理

测试中遇到"是 bug 还是设计意图"模棱两可的情况（例：删除章节是否需要二次确认、空状态文案是否合理、错误提示语气）：

- 由 Claude 按 **常识 + DESIGN.md + 同类产品惯例** 自行决策
- 所有自行决策记录到最终报告的"产品决策"小节
- 用户最后统一审核，不满意的再单独修改

## 6. 工具与环境

- **浏览器测试**：用 `/browse` 技能（gstack），不用 playwright MCP
- **AI provider**：DeepSeek（用户提供的临时 key），仅用于测试期间，不写入任何 commit / doc
- **测试数据**：临时项目 + 临时章节，测完清理
- **dev server**：测试前 `pnpm dev` 启动一次，结束后停止

## 7. 阶段任务概览

### 阶段 A：自动化测试 baseline

- 跑 `pnpm lint`、`pnpm test`、`pnpm test:e2e`
- 收集：失败用例 / 警告 / 单测覆盖率 / E2E 通过率
- 阻塞性失败先修，不阻塞的进入阶段 B 之后处理

### 阶段 B：浏览器穷举 QA（B3 范围）

覆盖矩阵：

| 模块 | 核心 | 边界 | 次要 |
|------|------|------|------|
| 项目 CRUD | 建/列/删/恢复 | 重复名/空名/超长名 | 设置弹窗 |
| 章节 | 建/写/排序/删 | 大文档/重命名/草稿 | 章节元信息、上下文菜单 |
| Tiptap 编辑器 | 输入/格式/撤销 | 1万字+/粘贴 HTML | 浮动工具条 |
| AI 对话 | 流式回复/工具建议 | key 错误/网络错/中断 | provider 切换 |
| 世界观 | 增/改/删/标签 | 关系互引/软删恢复 | 模板 |
| 关系图 | 拖拽/点击/AI 推荐 | 节点>50/孤立节点 | 布局快照 |
| 分析页 | 时间线/矛盾仪表 | 空数据 | — |
| 同步 | 入口可见 | 无 key 状态 | — |
| 导出 | 主格式 | 空章节 | — |
| 全局 | 路由/快捷键 | 移动端响应式 | 主题、深色 |

输出：bug 清单（id / 模块 / 严重等级 P0~P3 / 复现步骤 / 截图 / 期望 vs 实际）。

### 阶段 B 修复

- P0（崩溃 / 数据丢失 / 阻塞主流程）：必修
- P1（功能错误 / 严重体验问题）：必修
- P2（次要体验 / 提示文案）：尽量修
- P3（吹毛求疵）：仅记录到报告

### 阶段 C-perf：性能

- baseline：首屏 LCP / TTI、Tiptap 输入延迟（5k 字 / 50k 字）、关系图渲染（30 / 100 节点）、bundle 大小
- 优化方向：动态 import、memo、虚拟列表、tiptap 扩展按需加载、关系图渲染节流
- after：同指标对比，要求关键指标改进 ≥ 15% 或达到合理阈值

### 阶段 C-quality：代码质量

- 0 lint error（warnings 尽量清）
- 业务代码 `any` 类型清零（lib type 例外允许标注理由）
- 重复逻辑提取（仅当 ≥ 3 处重复且语义一致）
- 单文件 > 400 行的酌情拆分（仅在职责确实多于一项时）

### 阶段 C-a11y：可访问性 + DESIGN.md 合规

- 键盘可达：所有交互可 Tab / Enter / Esc 操作
- ARIA：dialog / menu / tooltip 角色齐全
- 对比度：正文 ≥ 4.5:1，UI 控件 ≥ 3:1
- DESIGN.md 红线：圆角 ≤ 8px、无阴影、无 pulse/shimmer/glow、字体合规、`lang="zh-CN"`

### 阶段 C-test：单测覆盖

- 关键模块补测（覆盖率目标按"所有公开 API + 关键分支"，不强求百分比）：
  - `src/lib/db/*`（meta、project、迁移）
  - `src/lib/ai/*`（client、prompts、suggestion-parser、tools/schemas）
  - 关系图算法（force-layout、布局快照）
  - 章节排序、软删除恢复

## 8. 最终报告结构

写入 `docs/superpowers/reports/2026-04-25-e2e-qa-and-quality-report.md`：

1. 概述（耗时、commit 数、bug 总数）
2. bug 清单（按模块）+ 修复状态
3. 性能 before / after
4. 代码质量数据（lint、any、文件长度）
5. a11y 检查结果
6. 单测覆盖增量
7. 产品决策记录（含理由）
8. 剩余 TODO（P3 + 不在范围的 enhancement）
9. 安全提醒（DeepSeek key 已在测试中临时使用，请用户 revoke）

## 9. 风险与缓解

| 风险 | 缓解 |
|------|------|
| QA 时间膨胀 | 每模块设上限 15 分钟，超时记 P3 跳过 |
| 修复引入新 bug | 每组 commit 后跑相关 vitest + 手动冒烟 |
| DeepSeek key 泄露 | 仅运行时使用，不写入文件，报告里强提醒 revoke |
| Tiptap 大文档卡死浏览器 | 用 5k / 20k / 50k 三档逐步压力测试 |
| 关系图大数据卡顿 | 测试上限 100 节点，超出仅记录而不强测 |

## 10. 完成标准

- 阶段 A / B / C 全部任务在 TaskList 中标记 completed
- 所有 P0 / P1 bug 已修复并验证
- 性能 before / after 数据齐全
- 最终报告已写入 `docs/superpowers/reports/`
- 所有改动已 commit（不 push，等用户确认）
